## Learned User Preferences

- Use Claude Code for autonomous worker flows, scripts, and documentation in this repo; do not substitute Codex for that role.
- Keep `CLAUDE.md` tracked in git so Claude Code can load project context on init; do not git-ignore files that are required for correct branching, worker behavior, or onboarding.
- Before destructive git commands that discard working tree changes, use stash, recovery paths, or explicit user confirmation; do not use `git reset --hard` casually when uncommitted work could disappear.
- Prefer minimal, explicit shell wrappers over large orchestration stacks unless added complexity clearly buys reliability here.
- Keep novel Parameter Golf experiments on dedicated `approach/<name>` branches when working manually, but land validated controller or repo-hardening changes on `main` and push `main` before considering the run finished.
- When the user asks to change git attribution on this machine (for example dropping Claude from co-authored trailers), follow that for local commits.
- For controller `claude -p` invocations in this repo, do not pin an explicit Claude model; use the CLI default/latest model selection.
- After substantive code or script changes, update the markdown that documents behavior so operators are not misled (only where such docs already exist and apply).
- Prefer ephemeral, deterministic worker executions (short-lived, schema-bounded AI sessions managed by a shell controller) over long-lived persistent sessions to prevent context drift and enforce tight safety boundaries.
- Delegate privileged operations like git push/fetch, pod selection, SSH, artifact collection, and infrastructure provisioning to the Mac shell controller via real checked-in control-plane scripts; RunPod pods should remain pull-only executors.
- Avoid executing free-form bash commands generated by the LLM (e.g., via eval); instead, require structured JSON output (like argument arrays) and construct execution commands safely in the shell wrapper.

## Learned Workspace Facts

- Use GitHub CLI for GitHub auth and Git credential helper: run `gh auth login` (HTTPS) and `gh auth setup-git` so `git push`/`git pull` to `github.com` use `gh` tokens; prefer `gh` for PRs and repo browser workflows; keep `origin` as HTTPS.
- RunPod usage for Parameter Golf should target pods whose names start with `pg-`; use full SSH via the pod's public IP for `scp`/`rsync` artifact sync, treat `RUNPOD_TEMPLATE_ID` as required Mac-side config for real launches, and avoid unrelated pods on the account without explicit user approval.
- Assume this machine is a controller-only environment for Parameter Golf: training and heavy experiment execution should run on RunPod, with GitHub used as the sync point for exact worker commits, and a local-only Claude loop is not sufficient proof that the Mac -> GitHub -> RunPod path is end-to-end ready.
- For unattended RunPod operation, schedule `scripts/runpod_reconcile.sh` periodically on the Mac controller; running it only at cycle start is not sufficient crash recovery.
- Detached or non-TTY launches of `claude -p` should feed non-interactive stdin (for example `< /dev/null`) so the process does not block waiting for terminal input.
- Default training layouts that write fixed artifact filenames under the repo root can collide when multiple runs share one checkout; isolate outputs per run when parallelism or overlapping experiments are possible.
- Use file locking (e.g., `flock`) when concurrent processes or autonomous runs write to shared ledgers (like `results.tsv` or spooled records) to avoid race conditions.
- When running `claude -p` with the `--bare` flag, auto-discovery of project configurations is skipped; you must explicitly pass `--settings` and `--mcp-config` to enforce project rules.
- For autonomous commits in controller flows, do not use broad `git add .`; stage only the relevant source and documentation files, exclude runtime directories like `worktrees/`, `experiments/`, and `registry/`, and do not suppress commit errors with `|| true`.
